# 航空模拟平台优化方向分析计划

## Summary

- 目标：基于现有 `AeroLab` Demo，对照“面向教学或科普的航空模拟平台”要求，整理当前项目还能继续优化的方向，并形成后续可执行的改进路线。
- 分析基线：当前项目已经满足“至少一种航空场景模拟 + 教学引导 + 实时反馈 + 数据可视化”的最低要求，且实际已覆盖基础飞行力学、风切变、单发失效、风洞流线、仪表认知训练等多个场景。
- 规划结论：后续优化不应再停留在“补齐有没有”，而应转向“教学深度、模型可信度、场景组织、可扩展性、可观测性、演示完成度”六个方向。
- 输出形式：后续执行阶段以“分层优化建议 + 对应落点文件 + 验证方式”为主，优先给出高价值、低风险、与现有代码结构兼容的改造方案。

## Current State Analysis

### 1. 项目结构与技术形态

- 根目录 `d:\2026\treaProject\README.md` 说明该项目为前后端分离 Demo，分为 `backend/` 与 `frontend/` 两部分。
- 前端位于 `d:\2026\treaProject\frontend`，使用 `Vue 3 + Vite + Three.js`，入口文件为 `frontend/src/main.js`，3D 场景在 `frontend/src/components/FlightScene3D.js`，飞行逻辑在 `frontend/src/sim/flightModel.js`。
- 后端位于 `d:\2026\treaProject\backend`，使用 JDK 自带 `HttpServer`，服务入口为 `backend/src/main/java/com/aerolab/AeroLabServer.java`，教学数据在 `TeachingGuide.java`，模拟计算在 `SimulationEngine.java`。
- 当前架构更偏“单页教学演示应用”，不是多模块平台型工程；前端状态集中，后端接口简单，适合继续演进但尚未抽象出平台层能力。

### 2. 已满足的需求点

- `frontend/src/sim/flightModel.js` 已支持基础飞行力学模拟：起飞、巡航、降落。
- 同文件中的 `flightPrograms` 已支持扩展教学脚本，包括 `wind-shear`、`engine-failure`、`stall-recovery` 等突发事件类内容。
- `frontend/src/main.js` 已包含仪表认知训练题、实时反馈文案、教学步骤、趋势图表、控制台、程序化示例切换。
- `frontend/src/components/FlightScene3D.js` 已提供 3D 飞机、场景切换、气流/力矢量可视化，因此“风洞气流可视化”和“为什么飞机能飞”的演示已有基础。
- `backend/src/main/java/com/aerolab/TeachingGuide.java` 与 `AeroLabServer.java` 已提供教学步骤、默认场景和单步更新接口，说明前后端教学链路已经打通。

### 3. 当前主要短板

- 教学内容虽然存在，但更多是“说明式展示”，缺少分层课程编排、学习目标、完成判定与结果记录。
- 飞行模型为简化公式模型，适合科普，但对“为什么机翼形状改变会影响升力/阻力”的因果解释还不够直观和可操作。
- 多场景虽然存在，但当前组织方式仍以程序按钮为主，尚未形成“模式入口 -> 目标 -> 操作 -> 反馈 -> 复盘”的完整教学流程。
- 前端主逻辑大量集中在 `frontend/src/main.js`，后续新增功能会继续增加耦合，影响维护和扩展。
- 后端当前更像演示辅助服务，缺少课程配置、结果记录、场景参数化等平台支撑能力。
- 缺少测试、性能监控、错误可视反馈和运行时诊断，对后续稳定迭代不利。

## Proposed Changes

### 方向一：把 Demo 升级为“教学流程完整”的课程体验

#### 涉及文件

- `d:\2026\treaProject\frontend/src/main.js`
- `d:\2026\treaProject\frontend/src/sim/flightModel.js`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/TeachingGuide.java`
- `d:\2026\treaProject\README.md`

#### 做什么

- 将当前“教学步骤 + 示例程序 + 提示文案”整合为明确的课程流：课程介绍、目标说明、阶段任务、完成判定、结果总结。
- 为每种场景定义“学习目标”和“达标条件”，例如起飞阶段要求在指定速度区间抬轮、巡航阶段要求稳定高度、风切变恢复要求避免失速。
- 增加结果页或复盘区域，解释用户在哪一步操作正确、哪一步导致速度损失或失速风险升高。

#### 为什么

- 这能更强地满足“必须明确服务于教学或科普”的要求。
- 现在的内容更像互动展示，课程化后更适合课堂演示、学生自学和成果展示。

#### 怎么做

- 在 `main.js` 中抽离课程状态与完成判定逻辑，不再只靠当前步骤文本提示。
- 在 `flightModel.js` 中补充每类场景的目标指标和评分输入字段。
- 在 `TeachingGuide.java` 中把教学步骤从静态提示升级为可配置课程元数据。
- 在 `README.md` 中明确课程模式、适用对象和操作路径。

### 方向二：强化“为什么会这样”的可解释可视化

#### 涉及文件

- `d:\2026\treaProject\frontend/src/components/FlightScene3D.js`
- `d:\2026\treaProject\frontend/src/sim/flightModel.js`
- `d:\2026\treaProject\frontend/src/main.js`
- `d:\2026\treaProject\frontend/src/styles/app.css`

#### 做什么

- 增强机翼形状变化、迎角变化、气流分离、失速边界等可视化表达。
- 为力矢量、升阻比、迎角、空速、垂直速度增加联动说明，让用户知道“某次操作导致了什么变化”。
- 增加关键时刻的冻结讲解或对比模式，例如“襟翼 0/1/2 档”对低速升力和阻力的对比。

#### 为什么

- 当前已经能看到气流和四力，但教学解释仍偏文字化。
- 要回答“为什么飞机能飞”或“为什么机翼形状会改变”，需要把抽象变量转成可观测的视觉因果。

#### 怎么做

- 在 `FlightScene3D.js` 中增加更清晰的气流状态层和构型变化动画。
- 在 `flightModel.js` 中暴露更多中间计算量，如升力系数、阻力系数、失速阈值、构型加成。
- 在 `main.js` 中新增因果提示卡片或对比面板。
- 在 `app.css` 中配套样式，保证信息密度提升后仍易读。

### 方向三：重组场景入口，形成真正的“平台”感

#### 涉及文件

- `d:\2026\treaProject\frontend/src/main.js`
- `d:\2026\treaProject\frontend/src/sim/flightModel.js`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/TeachingGuide.java`
- `d:\2026\treaProject\README.md`

#### 做什么

- 将现有场景划分为几个一级模式，例如“基础飞行力学”“风洞实验”“仪表认知训练”“应急处置训练”。
- 每个模式下提供场景说明、推荐操作、观察重点和适龄对象，而不是只暴露程序按钮。
- 让场景切换时同步切换教学文案、参数限制、图表重点和 3D 可视元素。

#### 为什么

- 当前项目功能上已不止一种场景，但入口仍偏开发演示风格。
- 模式化后更符合“平台 Demo”表述，也更方便答辩、汇报或后续扩展。

#### 怎么做

- 把 `flightPrograms` 从纯演示脚本扩展为“场景配置 + 教学配置 + UI 配置”的统一结构。
- 在 `TeachingGuide.java` 中输出按模式组织的元数据。
- 在 `main.js` 中引入一级导航和场景描述区域。

### 方向四：降低前端耦合，给后续扩展留空间

#### 涉及文件

- `d:\2026\treaProject\frontend/src/main.js`
- `d:\2026\treaProject\frontend/src/sim/flightModel.js`
- `d:\2026\treaProject\frontend/src/components/FlightScene3D.js`
- 可能新增：
  - `d:\2026\treaProject\frontend/src/modules/teaching/*`
  - `d:\2026\treaProject\frontend/src/modules/scenarios/*`
  - `d:\2026\treaProject\frontend/src/modules/instruments/*`

#### 做什么

- 拆分 `main.js` 中过于集中的状态、课程逻辑、交互逻辑和渲染逻辑。
- 把场景配置、教学步骤、图表转换、仪表训练和解释文案做模块化整理。
- 让 3D 层只负责展示，飞行计算层只负责状态更新，教学层只负责解释和流程控制。

#### 为什么

- 当前功能已进入“再加内容就会越来越难维护”的阶段。
- 如果后续真要加入无人机姿态控制、更多飞机类型或多人教学流程，现有结构扩展成本会迅速上升。

#### 怎么做

- 把场景元数据和教学元数据从 `main.js` 中移出。
- 给 `flightModel.js` 定义更清晰的输入输出边界。
- 明确组件职责，避免视图层直接承担太多业务推导。

### 方向五：提升后端的平台支撑能力

#### 涉及文件

- `d:\2026\treaProject\backend/src/main/java/com/aerolab/AeroLabServer.java`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/TeachingGuide.java`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/SimulationEngine.java`
- `d:\2026\treaProject\backend\README.md`

#### 做什么

- 将后端从“静态步骤 + 单次计算”扩展为“场景配置中心 + 课程配置中心 + 结果记录入口”。
- 统一返回课程、场景、推荐参数、完成条件等结构化数据，减少前端硬编码。
- 为后续保存用户练习结果、生成教师讲解材料或接入更多场景预留接口。

#### 为什么

- 当前后端价值较弱，很多教学数据仍在前端本地定义。
- 如果要体现平台属性，后端应该开始承担配置与内容编排职责。

#### 怎么做

- 在 `TeachingGuide.java` 中把步骤、场景、推荐参数统一建模。
- 在 `AeroLabServer.java` 中补充课程/场景类接口。
- 在 `SimulationEngine.java` 中考虑暴露更多可解释指标，而不是只返回最终状态。

### 方向六：增强教学评估、可观测性与稳定性

#### 涉及文件

- `d:\2026\treaProject\frontend/src/main.js`
- `d:\2026\treaProject\frontend/src/sim/flightModel.js`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/AeroLabServer.java`
- `d:\2026\treaProject\backend/src/main/java/com/aerolab/SimulationEngine.java`
- 可能新增测试文件与验证脚本

#### 做什么

- 增加关键场景的结果校验、边界提示、异常状态展示和操作日志。
- 为典型场景建立最小验证集，如起飞是否能稳定离地、风切变恢复时是否会因为错误迎角进入高失速风险。
- 对前后端同构计算逻辑建立对比检查，减少结果偏差。

#### 为什么

- 当前项目是教学 Demo，但教学演示也需要稳定和可复现。
- 没有验证与诊断时，功能越多，越难保证演示质量。

#### 怎么做

- 围绕 `updateFlight` 与 `SimulationEngine.update` 设计对照校验点。
- 为关键状态增加更明确的 UI 提示和错误反馈。
- 后续若进入执行阶段，再按价值补充最小测试。

### 优先级建议

- P0：教学流程完整化、可解释可视化增强。
- P1：场景入口重组、前端模块化拆分。
- P2：后端平台化、教学评估与测试支撑。

## Assumptions & Decisions

- 假设本次目标是“基于当前代码提出优化方向并形成后续实施计划”，而不是立刻修改业务代码。
- 假设该项目的主要使用场景为课堂演示、科普展示或课程作业答辩，因此教学表达优先级高于物理严谨度。
- 决策上优先沿用现有技术栈：前端继续基于 `Vue 3 + Vite + Three.js`，后端继续沿用轻量 Java HTTP 服务，避免为了“平台化”立刻引入过重框架。
- 决策上把“优化方向”分为产品体验、教学设计、代码架构、后端支撑、验证稳定性五类，避免只从视觉或代码层面给建议。
- 若进入执行阶段，优先做不破坏现有演示链路的渐进式重构，而不是一次性大改。
